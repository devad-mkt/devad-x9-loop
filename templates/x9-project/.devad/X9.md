# Devad X9 Project Router

X9 is the simplified successor to the failed X7 manager-control experiment.
It keeps X5 as the base: one repo-local router, one active task, one routed
rule pack, one handoff. X9 adds only the useful X7 idea: a user-managed
multi-chat merge discipline for parallel work.

## Operating Model

- The user is the manager.
- Codex chats are normal workers, not a swarm controlled by another chat.
- A manager chat may draft prompts, inspect worker reports, review diffs, and
  recommend merge order, but it must not run an always-on observe/control loop.
- Workers work like X5: narrow task, narrow files, local verification, concise
  handoff. They do not read broad `.devad`, old proof trees, old chats, or
  other workers' files.
- `.devad` is project reference memory, not disposable noise. Task-related
  `.devad` docs, rules, handoffs, plans, prompts, proof summaries, and ledgers
  must be committed and pushed with the code they explain.

## Authority

```text
system/developer/tool rules -> newest user request -> source/runtime evidence -> .devad/X9.md -> .devad/ACTIVE.md + current HANDOFF -> routed .devad/rules pack -> assumptions
```

If docs conflict with source, browser, tests, Git, deploy output, or runtime
evidence, treat docs as stale. Update docs only when the task requires it.

## Commit Discipline

- Include the relevant `.devad` files in every implementation commit when they
  describe the task, plan, proof, risk, handoff, routing, or deploy state.
- Do not blindly stage the entire `.devad` tree. Stage the named task-related
  files and report any remaining unstaged `.devad` files with a reason.
- A final answer must say whether `.devad` was updated and uploaded, unchanged,
  or left with unstaged task-related files.
- Never commit secrets, raw provider responses, cookies, OAuth codes, `.env`
  values, full deploy JSON, or token-bearing logs into `.devad`.

## Branch Source Policy

- Use `<project-root>` on
  `<implementation-branch>` as the X9 development
  source for new CORE POST work.
- Treat `<deployment-branch>` as the current
  Dokploy/live deploy branch only. Do not start new feature work there.
- A live hotfix may be applied to the deploy branch only when the task
  explicitly needs production proof or deploy recovery. After any deploy-branch
  hotfix, immediately reconcile the same change back into `core-x9`, verify it,
  and push the X9 branch before continuing feature work.
- A deploy/live-proof request is not complete if Dokploy is still serving a
  branch that lacks the implementation. Use the deploy closure gate in
  `.devad/rules/x9-branch-deploy-policy.md`: controlled deploy bridge, explicit
  owner branch switch, or blocked handoff.
- `<source-root>` is not the clean X9 source while it has unrelated
  local changes. Use it only deliberately for deploy-branch inspection or after
  a fresh cleanliness check.

## Deploy Bridge Drift Guard

Before coding, classify the checkout:

| Classification | Meaning | Allowed action |
| --- | --- | --- |
| `X9_SOURCE` | `<project-root>` on `<implementation-branch>` | normal X9 work |
| `DEPLOY_BRIDGE_ACTIVE` | clean v105 bridge made only to apply already committed X9 work | cherry-pick/reapply committed X9 commit, verify, push/deploy |
| `DEPLOY_BRIDGE_LOCKED` | bridge after push/deploy/proof | inspect or verify only |
| `ORPHAN_PATCH` | dirty bridge/v105/detached checkout with uncommitted work | export/report diff, then port to X9 or discard |

If the path contains `deploy`, `bridge`, `v105`, or `x9w\v105`, or Git reports
detached HEAD or branch `<deployment-branch>`, stop
before coding unless the task is strictly deploy inspection, diff export,
deploy proof, or applying a previously committed X9 commit. New debug, feature,
or docs work must restart from `<project-root>`.

## Startup

For nontrivial work:

1. Identify repo root, branch, upstream, HEAD, dirty files, and whether
   `.devad/ACTIVE.md` matches this checkout.
2. Read `.devad/X9.md`.
3. Apply the Deploy Bridge Drift Guard. If this is not `X9_SOURCE`, do not
   implement new work. Report the classification and next safe source folder.
4. For branch, resume, deploy, hotfix, handoff, or source-selection work, run a
   fresh Git remote check and load `.devad/rules/x9-branch-deploy-policy.md`.
   Exact SHAs beat GitHub UI timestamps, old handoffs, and `.devad/ACTIVE.md`.
5. For deploy or live-proof work, verify the Dokploy app branch/latest
   deployment safe fields through `.devad/rules/devad-deploy-dokploy.md` and
   `.devad/tooling/dokploy/HOW-TO-USE.md` before choosing a source branch.
6. Read `.devad/ACTIVE.md` and the current feature `HANDOFF.md` only for
   branch, resume, deploy, merge-risk, or explicit handoff work.
7. If the task is broad/noisy, or you are unsure which Devad/XCO/Laravel pack
   applies, read `.devad/rules/x9-read-policy.md` once, then choose.
8. For user-named URLs, visible UI behavior, deploy/live proof,
   provider/platform proof, or API/CLI/MCP proof, load
   `.devad/rules/x9-acceptance-proof.md` and create the acceptance matrix before
   coding.
9. Load only the routed rule pack below. Add a second domain pack only when
   touched files cross domains.
10. If coordinating multiple chats, load `.devad/rules/x9-manager.md`.
11. If external model help is needed, load `.devad/rules/devad-sidecar-models.md`
   and `.devad/rules/devad-headless-cli-sidecars.md`.

Do not read `.devad-dont-read`, `.devad/features`, old reports, old chat
exports, X7 coordination/control files, or broad logs unless the current task
gives a narrow reason. Do not load every rule pack or reread rule files after
each tool call.

## Optional Feature Reference Packs

Feature folders are not startup memory. Read only the task file when it exactly
matches the current lane, then at most one linked section file:

| Task signal | Optional reference |
| --- | --- |
| POST API/CLI/MCP proof runner, provider proof payloads, API-first live proof | `.devad/features/post-mcp-cli-api--2026-06-22/TASK.md` |
| POST platform expansion, tenant-credential platforms, provider adoption plan | `.devad/features/post-platfroms-expansion-2026-06-22/execution-plan-full/TASK.md` |

Never load either full feature folder by default.

## X9 Parallel Work Rule

Use parallel chats only when work can be split by owned file/path lanes. Prefer:

```text
<worktree-root>\<lane>
```

Each worker must report:

```text
Worktree:
Branch/HEAD:
Task:
Acceptance matrix:
Files touched:
.devad files staged/uploaded:
Tests run:
Visible proof matrix:
Known risks:
Ready to merge? yes/no
```

The user or manager chat combines one worker at a time. A worker PASS is not a
merge PASS, deploy PASS, or browser PASS.

## Retired From X9

- X7 manager-control observe/poll loops.
- ACP transports for OpenCode/Cursor/PI.
- PI workflow execution as a dependency.
- OpenRouter routes/model IDs.
- Automatic broad reads of `.devad-dont-read`, old proofs, or old chat logs.

## Output Contract

Start with `TLDR:`. Prefer compact tables. Before coding, report repo identity,
branch/HEAD, dirty state, and routed packs when risk is nontrivial. Final
answers must separate changed, verified, skipped, blocked, and remaining owner
action, and must state the `.devad` upload status for task-related docs.
